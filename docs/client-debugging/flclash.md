# FlClash

Launch path:

- Runner: `ProxyAppClientRunner`
- Harness entry: `createClientRunner({ client: "flclash" })`
- Runtime config: Mihomo YAML generated by `buildMihomoRuntimeConfig` or
  protocol-specific Mihomo builders.

Debug surface:

- Primary debug adapter: `VmBridgeDebugClient`
- Source: Flutter VM service exposed by `ProxyAppManager`
- Runtime tweaks: extension inventory only; no stable protocol-mutation API
  comparable to Clash controller endpoints.

Testing integration:

- GUI app is launched headless through `ProxyAppManager`.
- Harness waits for the locally exposed proxy port before running traffic.
- Protocol coverage comes from the same reusable scenarios used by
  `clash-verge-rev`, but GUI automation is a smaller subset than raw Mihomo-core
  execution.

Artifacts:

- `debug-initial.json`
- `debug-tweaks.json`
- `debug-final.json`
- `report.json`

Client-side limitations:

- `vless_kcp`: Mihomo VLESS path still falls through to a TCP dial when
  `network: mkcp` is configured. Treat as client/runtime gap, not `wrongsv`.
- `vless_quic`: still unconfirmed in the current GUI automation path.
- The top-level FlClash repo is fork-pushable on `debug`, but the actual Mihomo
  transport code lives in the nested `core/Clash.Meta` git repo. Do not record
  a new submodule SHA in FlClash unless the nested repo commit is also pushable.
